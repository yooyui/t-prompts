# 模块：标准工作流程

## 五阶段工作流

### 阶段一：深入理解与分析 (10-15%)
**目标**：全面掌握现状，识别改进方向

**执行清单**：
- [ ] 审阅提供的资料/代码
- [ ] 理解核心业务逻辑
- [ ] 识别技术架构和依赖
- [ ] 发现痛点和瓶颈
- [ ] 评估技术债务

**输出物**：
- 现状分析文档
- 问题清单
- 初步改进建议

### 阶段二：目标规划 (15-20%)
**目标**：制定清晰、可衡量的方案

**执行清单**：
- [ ] 明确任务范围
- [ ] 定义验收标准
- [ ] 制定技术方案
- [ ] 风险评估
- [ ] 时间估算

**决策模板**：
```markdown
## 技术决策
- **选项A**：[描述] | 优点：... | 缺点：...
- **选项B**：[描述] | 优点：... | 缺点：...
- **最终选择**：[选项] | 原因：...
```

### 阶段三：分步实施 (50-60%)
**目标**：逐步落地改进方案

**实施策略**：
1. **分解任务**
   - 拆分为 2-4 小时的子任务
   - 明确依赖关系
   - 设置里程碑

2. **迭代开发**
   ```
   循环 {
     编码 → 测试 → 验证 → 提交
   }
   ```

3. **进度追踪**
   - 每完成一个子任务更新进度
   - 遇到阻塞立即记录
   - 及时调整计划

### 阶段四：验证与质量保证 (10-15%)
**目标**：确保交付质量

**验证检查表**：
```markdown
## 质量门禁
- [ ] 单元测试通过率 > 95%
- [ ] 代码覆盖率 > 80%
- [ ] 无高危安全漏洞
- [ ] 性能基准达标
- [ ] 代码审查完成
- [ ] 文档已更新
```

**回归测试策略**：
1. 运行自动化测试套件
2. 手动测试关键路径
3. 边界条件验证
4. 错误恢复测试

### 阶段五：总结与展望 (5-10%)
**目标**：提供清晰的汇报

**总结报告模板**：
```markdown
# 任务总结报告

## 完成情况
- **核心成果**：[列表]
- **交付物**：[列表]
- **指标达成**：[数据]

## 技术亮点
- [创新点或优化点]

## 经验教训
- **成功经验**：[可复用的经验]
- **踩坑记录**：[需要避免的问题]

## 后续建议
- **立即可做**：[低成本高收益]
- **计划改进**：[需要规划的改进]
- **长期优化**：[架构级别的优化]
```

## 快速执行模式

### 小任务模式 (< 2小时)
```
1. 快速理解 (10分钟)
2. 直接实施 (1.5小时)
3. 验证提交 (20分钟)
```

### 中等任务模式 (2-8小时)
```
1. 需求分析 (30分钟)
2. 方案设计 (30分钟)
3. 分步实施 (5-6小时)
4. 测试验证 (1小时)
```

### 大任务模式 (> 8小时)
```
需要加载: /load task-management
使用分层递进式架构处理
```

## 并行工作策略

### 可并行的任务
- 独立模块开发
- 不同文件的修改
- 测试用例编写
- 文档更新

### 必须串行的任务
- 有依赖关系的功能
- 数据库迁移
- 破坏性变更
- 部署相关操作

## 风险管理

### 风险等级
- 🔴 **高风险**：立即处理
- 🟡 **中风险**：制定缓解计划
- 🟢 **低风险**：持续监控

### 风险应对策略
1. **规避**：改变方案避免风险
2. **缓解**：降低风险概率或影响
3. **转移**：将风险转移给第三方
4. **接受**：承认并准备应对
